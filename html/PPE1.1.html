Contexte

Le site a pour but de mettre en relation un vendeur de produit bio ou non qui n’est pas un professionnel et un acheteur pour ce produit qui se servirait de la géolocalisation et d’une map afin de trouver le meilleur vendeur au plus proche de sa position. Un système d’utilisateur Contrôleur qui aurait en échange de ses contrôles chez les vendeurs des réduction lors de ses achats contrôlerait la qualité des vendeurs mais aussi des produits pour les certifier bio ou non. Un système de commentaire serait mis en place, ainsi qu’un système de message et de report. Le système de paiement se ferait exclusivement via le site web.

En groupe, nous devions créer un site web avec le langage PHP en nous aidant du framework Laravel dans un style de programmation MVC. La partie back-end devait être une API écrite via le framework Laravel. La partie base de données serait un SGBD MySQL. Nous devions aussi concevoir un mini cahier des charges préalablement afin de dresser les différents besoin. La partie front-end nous était laissé au choix.

Réalisation

Dans un premier temps Nous avons fait le choix de réalisé le cahier des charges afin d’y voir plus clair à travers les différentes contraintes du projets. Nous avons par la suite, en groupe décidé de réalisé la base de données qui se fait en Laravel via des fichiers de Migration. Nous avons alors utilisé la méthode Merise afin de dresser nos différents tables et les relations qui les lient entre elles. Nous avons décidé d’utiliser github pour le versionning et blade pour le front qui est l’outil de template de Laravel S’en est suivi alors la répartition des fonctionnalités, les fonctionnalités que j’ai eu à gérer ont été les suivantes :

  1. Gestion des utilisateurs
  2. Gestion de la sécurité (qui a accès à quel ressource)
  3. Conception d’une grande partie de l’API
  4. Mise en place du mailing pour certaines fonctionnalités

J’ai donc conçu la gestion des utilisateurs sur l’api vi un token d’authentification et un username(email) qui me permettait donc d’authentifié l’utilisateur via cette paire token email.

La gestion de la sécurité c’est fait avec un grand nombre de middleware en fonction du statut de connexion des utilisateurs dans la session. Un traitement est fait à chacune de leur requête http qui le nécessite afin de voir si oui ou non ils peuvent avoir accès à la page ou information demandée. Si ce n’est pas le cas une redirection de l’utilisateur sur une autre route est alors effectué.

La partie API fut une partie très intéressant car il a fallu que je me documente pour réaliser tout la partie Modèle dans l’ API ainsi que la partie Controller afin que le site web n’est qu’à gérer les vues et les redirection entre les pages si nécessaire. Toutes les requêtes si elles ont besoin d’un contrôle doivent être envoyé à l’API via le protocole http, la requête et les informations qu’elle contenait doivent être récupéré, traité puis un JSON est renvoyé au site web avec un statut 400 et des messages d’erreur si quelque chose ne va pas et un statut 200 si au contraire c’est un succès.

La mise en place du mailing a été fait via le framework Laravel ainsi que le fichier d’environnement .env dans lequel l’adresse du serveur smtp est renseigné. Ce smtp permet de rediriger les mails vers le destinataire. Dans notre cas nous avons utilisé la sandbox de mailtrap.io afin de pouvoir voir tous les mails envoyé et reçus comme si ceux-ci était vraiment envoyé. Un service a été fait afin d’y faire appel pour envoyé des mail via des templates blade.

Pour finir j’ai fait de l’intégration sur le front-end de certaines fonctionnalité comme le changement de statut utilisateurs, l’ajout de nouveaux statuts utilisateur, la gestion administrateur minimal des report et contacts. Pour Conclure toute les contraintes ont été remplis et le site était fonctionnel
